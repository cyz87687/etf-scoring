# ETF 指数四维量化评分模型 · 优化建议与执行报告

> 版本：基于原 V5/V4 体系重构至 **V6**（模型名沿用 `index_v5.html` / `v5_data.json` 文件名约定）
> 交付目录：`/workspace/etf-scoring/`
> 目标：在**免费 API 限制**下，把"能跑但结果失真"的评分模型，修复为"结果诚实、可落地、可自动化"的版本。

## 一、项目诊断（熟悉项目后发现的真问题）

原项目由 7 个文件构成：`build_v4.py`、`fetch_data.py`、`index_v5.html`、`skill.md`、`v5_data.json`、`指数数据.xlsx`、评分体系文档。通读后定位出 **5 类问题**，按严重程度排序：

| # | 问题 | 严重程度 | 后果 |
|---|------|---------|------|
| 1 | **指数宇宙不一致**：`build_v4.py` 用 Excel 的 30 个指数，`fetch_data.py` 硬编码 50 个独立指数 + 独立 `ETF_MAP` | 🔴 致命 | `--api` 模式下 `quote.get(code)` 永远取不到数据，覆盖率崩塌（PB/PS/股息/动量/板块/ETF 申赎/新闻全为 0） |
| 2 | **`pct_scores` 虚假区分度**：用顺序枚举赋秩，全 0 / 全相等值被人为拉伸到 0–100 | 🔴 致命 | 资金面/消息面"全字段为 0"却被排出 0–10 的假高低分（如 `capital=9.3` 实为全 0 假排名） |
| 3 | **置信度虚高**：CSI 指数技术面 RSI/波动率没取到却用 5 分填充，综合置信度被抬到 0.58 | 🟠 高 | 低数据指数被误排高位，报告不可信 |
| 4 | **质量因子稀疏失真**：科创芯片仅 PE=301 一项，单因子排末位直接得 0 分 | 🟠 高 | 单指标极端化，子因子覆盖不足时分数无意义 |
| 5 | **HTML 与配置/CI 缺漏**：CSS 直接大段写进 f-string 易转义出错；`skill.md` 引用的 `daily-update.yml` 实际缺失 | 🟡 中 | 报告维护困难；项目无法自动更新 |

> 补充事实：原 `fetch_data.py` 把估值主源押在**东方财富**，而东方财富 API 在当前沙箱被网络拦截（连接重置），即便不修宇宙不一致问题，估值也会整体失败。

## 二、优化建议（已分层落地）

### 建议 1 · 架构层：统一指数宇宙（消除 #1 致命 bug）
- **原则**：单一配置来源。以 `指数数据.xlsx` 为唯一真源，所有模块 `import` 同一份 `INDICES` / `ETF_MAP` / `INDEX_TO_SECTOR`。
- **落地**：`fetch_data.fetch_all(indices, etf_map)` 由调用方传入 universe，不再内部硬编码。

### 建议 2 · 数据层：多源容错 + 真实技术指标（解决数据缺口）
- **多源容错**：东方财富（估值/ETF 资金流/新闻）为主，腾讯（行情/换手率/K 线）兜底，新浪（板块）补消息面。任一源失败不影响整体。
- **K 线真实指标**：对腾讯有 K 线的 9/30 指数，真实计算 `RSI(14)`、年化波动率、`60 日动量`、`120 日价格分位`、`10 日量价比`。
- **诚实标记**：CSI 指数（约 20 个）在免费 API 下无行情/无 K 线，直接标记为"无数据"，不伪造 5 分。

### 建议 3 · 评分引擎：修排名 + 加置信度（解决 #2/#3/#4）
- **`pct_scores` 重写**：等值取**平均秩**；`None`（缺失）→ 中性 50 分，不再被当成有效值参与排名。
- **资金面/消息面零值转 `None`**：避免"全 0 假排名"。
- **质量因子稀疏收缩**：子因子数 `<3` 时向中性 5 收缩（`shrink = min(1, len/3)`），单指标不再极端化。
- **置信度折扣 `adj_total`**：综合置信 `<60%` 时最多扣 2 分，把低数据指数压到排序低位；置信 `<35%` 额外标注"数据不足·仅供参考"。

### 建议 4 · 报告层：模板化 + 置信度可视化（解决 #5）
- **`build_report.py` 独立模块**：用占位符模板替代 f-string 拼接，杜绝转义残留（已校验无 `__xxx__` 占位符遗漏）。
- **新增展示**：置信度徽章、排序分（adj_total）、数据不足标注、覆盖率统计面板（每行因子可见覆盖率）。

### 建议 5 · 自动化：补全 CI（解决 #5 的 CI 缺失）
- 新增 `.github/workflows/daily-update.yml`：交易日 15:30（北京时间，`30 7 * * 1-5` UTC）跑 `build_v4.py --api` 并部署 GitHub Pages，支持手动触发。

## 五、必须坦白的现实约束（非代码缺陷）

当前沙箱网络环境下，**免费 API 的真实覆盖率有限**，这是环境限制而非代码 bug：

| 数据源 | 沙箱状态 | 影响 |
|--------|---------|------|
| 东方财富 push2 | ❌ 被拦截（连接重置） | 估值(PB/PS/股息)、ETF 资金流、新闻基本为 0 |
| 腾讯财经 qt.gtimg.cn | ✅ 可用 | 仅 sh/sz/hk/ETF 有行情；CSI 指数无覆盖 |
| 腾讯 K 线 web.ifzq | ✅ 可用 | 仅 9/30 指数有 K 线 → 真实技术指标仅覆盖 9 个 |
| 新浪板块 | ✅ 可用 | 板块涨跌可获取 |

**结果**：实测覆盖率 PE 8/30、技术面 9/30、板块 11/30、新闻 0/30。模型的"稀疏收缩 + 置信度折扣 + 数据不足标注"正是为这种**数据残缺场景**设计——它不会伪造分数，而是如实降级。

**要拿到完整数据（PB/PS/股息、北向、融资余额、CSI 全量），需要：**
1. 在**可访问东方财富/专业数据商**的环境部署（本机或 CI runner 网络可达）；
2. 或接入 `westock-data`（非沙箱可用）等本地数据终端；
3. 或替换数据源为 AkShare / Tushare（需 token）等更完整的免费库。
